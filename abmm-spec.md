# AI Background Music Maker (ABMM) 仕様書

対象: vibe coding（AIコーディングエージェントによる実装）用のプロジェクト仕様書
バージョン: v0.5
最終更新: 2026-07-12

---

## 1. 概要

**AI Background Music Maker（略称: ABMM）**は、自然言語での雰囲気指定＋スライダー調整で
**著作権フリーのインストゥルメンタルBGM（ボーカルなし）**を生成できるApple Silicon Mac専用デスクトップアプリである。

- 入力: 自然言語（例:「雨上がりの午後、少し切ないローファイジャズ」）＋ スライダー微調整
- 出力: 1〜60分の間で長さを指定できるBGM（WAV/MP3）
- ボーカル・歌詞は一切生成しない（インスト限定）

### 想定ユースケース
街歩き・散歩系YouTube動画のBGMは代表的な利用例の一つであり、それ以外にも以下のような
「著作権を気にせず使えるBGMが欲しい」場面全般を想定する。

- 個人の趣味の動画・Vlog
- 会社の社内イベント・展示会等で流す映像のBGM
- プレゼン資料・発表動画の背景音楽
- その他、汎用的な著作権フリーBGMが必要な用途全般

### 想定ユーザー
動画やイベントごとに雰囲気の異なるロイヤリティフリーBGMを、自分の言葉でサッと作りたい人（YouTuber、個人クリエイター、企業の広報・イベント担当者など）。

## 2. 設計方針（重要）：2フェーズ生成アーキテクチャ

音楽生成を**「作曲フェーズ」と「レンダリングフェーズ」の2段階**に明確に分離する。
3DCGでいう「モデリング（作曲）」と「レンダリング（画づくり）」の関係に近い。

| フェーズ | 役割 | 使用技術 |
|---|---|---|
| **Phase 1: 作曲フェーズ** | 自然言語＋スライダー指示から、曲の構成・楽器・音符データ（MIDIステップデータ）をLLMが直接生成する | ローカルLLM（Ollama等）が構造化MIDIデータをJSONで出力 |
| **Phase 2: レンダリングフェーズ** | Phase 1で生成されたMIDIデータを実際の音声波形に変換する | 生成AI/ニューラルモデルによる音声合成。PCスペック・用途（プレビュー/本番）に応じてモデルとパラメータを選択可能 |

この分離により、以下のメリットがある。

- 作曲結果（MIDI）はテキスト/JSONとして保存・再利用・**将来的な手動編集**が可能
- レンダリングは重い処理のため、PCスペックや用途（プレビュー確認用か、最終書き出し用か）に応じてモデル・パラメータを切り替えられる
- 同じMIDIデータを別の音源設定・別のレンダリングモデルで**再レンダリングし直す**ことができる（作り直しのコストが低い）

### Phase 1（作曲）の詳細方針
LLMに「テンポ・キー・楽器編成・雰囲気」などの抽象パラメータだけを出力させるのではなく、
**曲の構成（イントロ/展開/アウトロ等）と、各トラックの音符（ステップ入力データ）まで直接生成させる**。
自然言語での指示（例:「イントロは静かなピアノソロ、途中からアコギとパッドが入って盛り上がる」）と、
スライダーで指定した数値（テンポ・エネルギー・曲の長さ等）の両方をプロンプトに含めて、LLMに構造化MIDI JSONを出力させる。

### Phase 2（レンダリング）の詳細方針
Phase 1で確定したMIDIデータを、生成AIモデルを使って音声化する。
実行しているMacのスペック（チップ世代・メモリ量など）を自動検出し、複数のモデル（軽量/標準/高品質）から選択できるようにする。
「数十秒だけの試し聴き（プレビュー）」と「全体の本番書き出し」で、それぞれ別のモデル・パラメータを使い分けられるようにする。
モデルの実体は差し替え可能な抽象化レイヤー（レンダラーインターフェース）として設計し、
オープンソースの音声生成モデルのエコシステムの進化に応じて後から入れ替えられるようにする。

## 3. 動作環境

- **OS**: macOS、**Apple Siliconのみサポート**（Intel Macは非対応）
- **Python**: 3.11以上
- **ローカルLLM実行環境**: [Ollama](https://ollama.com/)、および Apple Silicon向けに最適化された`MLX`ベースのモデル実行系も候補として調査する
- ネットワーク接続: 不要（完全オフライン動作が目標。モデルは初回セットアップ時のみDLが必要）

Apple Siliconのみに絞ることで、統合メモリ構成やMLX等のApple Silicon最適化フレームワークを前提にした設計・スペック判定ロジックをシンプルにできる。

## 4. 技術スタック

| レイヤー | 技術 | 理由 |
|---|---|---|
| 言語 | Python 3.11+ | 音楽理論・音声処理ライブラリが充実。単一言語でvibe codingしやすい |
| デスクトップUI | `pywebview`（HTML/CSS/JSをネイティブウィンドウで表示） | Electron/Node不要で軽量。Mac専用アプリとして`.app`化しやすい |
| フロントエンド | 素のHTML + CSS + JavaScript（フレームワーク不要、必要ならAlpine.js程度） | 小規模UIなのでビルドチェーン不要にして実装コストを下げる |
| バックエンドAPI | FastAPI（`pywebview`から`localhost`で呼び出す、もしくは直接Python関数呼び出し） | 将来Web版に展開する場合も流用しやすい |
| Phase 1（作曲）LLM | Ollama + オープンソースLLM（`llama3.2:3b` または `gemma2:9b`など） | ローカル実行・無料・オフライン。JSON構造化出力に対応できる程度の指示追従性があるモデルを選定 |
| Phase 2（レンダリング）モデル | 複数候補をプラガブルに切替可能な抽象化レイヤー（詳細は10章） | 現時点でMIDI条件付き音声生成のOSSモデルは発展途上のため、実装フェーズで比較検証しつつ差し替え可能な設計にする。Phase 1と同一モデルを流用することも、専用の別モデルを使うことも可能な設計とする |
| MIDIデータ⇔ファイル変換 | `pretty_midi` または `mido` | LLM出力のJSON（ステップデータ）と標準MIDIファイル(.mid)を相互変換 |
| 簡易プレビュー用シンセ | `pyfluidsynth` + オープンライセンスSoundFont | Phase 1完了直後の即時プレビュー、およびPhase 2の軽量フォールバックとして使用 |
| 音声処理・書き出し | `numpy`, `soundfile`, `pydub`（クロスフェード・ループ処理、WAV/MP3書き出し） | ループ生成・フォーマット変換に必要 |
| ハードウェア検出 | `platform`, `psutil`、Apple Siliconのチップ判定（`sysctl`呼び出し等） | Phase 2のモデル選択・生成可能な曲の長さ上限判定に使用 |
| パッケージング | `pyinstaller` または `py2app` | 配布用の`.app`バンドルを作成 |

## 6. 機能要件

### 6.1 自然言語＋スライダーによる作曲指示（Phase 1入力）
- テキスト入力欄に、雰囲気だけでなく**曲の構成・展開・楽器指定**まで自由記述できる
  例:「イントロは静かなピアノソロ、Aメロでアコギが加わり、サビ的な盛り上がりでパッドとソフトドラムが入る。全体的に温かく穏やかな午後の雰囲気」
- スライダーで指定した数値（テンポ・明暗・エネルギー・密度・楽器バランス・**曲の長さ**等、詳細は11章）も同時にプロンプトへ組み込む
- LLMはこれらを踏まえて、8章で定義するMIDI JSONスキーマに従った構造化データを出力する

#### 6.1.1 Ollama接続確認とLLMモデル管理
- アプリ起動時および定期的なポーリング（例：10秒毎）により、ローカル環境の Ollama 接続状況を検出する。
- 接続が確認できない場合、メインUIに「Ollama未起動」警告を表示し、作曲ボタンを無効化する。
- Ollamaが起動しているが、選択されたLLMモデル（デフォルト：`llama3.2:3b`）がダウンロードされていない場合、モデル選択欄で「未ダウンロード」警告を表示し、「モデルをダウンロード」ボタンを提供する。
- ダウンロード時はリアルタイムに進捗バーを表示する。ダウンロードが完了するまで作曲ボタンは無効化される。


### 6.2 曲の長さ指定（1〜60分）
- 曲の長さをスライダーで1分〜60分の範囲で指定できる
- **PCのスペックに応じて、指定可能な長さの上限を自動調整する**（例: メモリが少ない/非力なチップの場合、レンダリング可能な上限を短めに制限し、スライダーの可動域自体をその上限までに制限する）
- 上限を超える長さを指定しようとした場合は、UI上でスライダーの可動域を制限するか、警告を表示して調整を促す
- Phase 1で生成するMIDIの長さそのものを指定長に合わせて生成する（短い場合はセクションを追加生成、または後段でループ処理により尺を伸ばす方式も検討）

### 6.3 LLMによる高次構成プラン（Chord Plan）生成とPythonによるMIDI音符展開（Phase 1出力）
- LLMは、文脈制限や文法エラーを回避するため、細かな音符列ではなく、高次構成プラン（Chord Plan）であるキーモード、スタイル、テンポ、楽器配置を含んだ簡潔なJSON構造を生成する。
- Python側の展開エンジン（`expand_chord_plan_to_midi`）がこのプランを読み込み、音楽理論に沿った多様なコード進行プールから自動的にセクション構成（Intro ➡ Verse ➡ Chorus ➡ Outro）と高度なMIDI音符データを展開・生成する。
- 毎回異なる響きにするため、生成時にランダムなキー転調（半音単位のオフセット）を自動適用する。
- 出力は`pretty_midi`/`mido`で標準MIDIファイル(.mid)に変換され、内部的に保持される。
- パース失敗・スキーマ逸脱時は自動リトライ（プロンプト修正）を行い、それでも失敗する場合はデフォルトのコード構成テンプレートからフォールバック展開する。

### 6.4 Phase 1 簡易プレビュー再生
- Phase 1で生成されたMIDIを、`pyfluidsynth`＋SoundFontで即座に音声化し、プレビュー再生できる
- 曲全体の構成を素早く確認するための機能で、Phase 2の重い処理を待たずに骨格を確認できる

### 6.5 Phase 2 プレビューレンダリング（本番書き出し前の試し聴き）
- 本番の全尺レンダリングを実行する前に、**数十秒（例: 30秒程度）だけをPhase 2のモデルで実際にレンダリングして試聴できる**
- プレビュー用と本番用で、使用するモデル・パラメータを個別に設定できる（7.7節参照）。例えば「プレビューは軽量モデルで素早く確認し、本番は高品質モデルで書き出す」といった使い分けが可能
- プレビューの結果に満足すれば、そのまま本番の全尺レンダリングに進む

### 6.6 Phase 2 本番レンダリング・モデル/パラメータ選択
- 確定したMIDIデータを、生成AIモデルを使って本番用の音声にレンダリングする
- アプリ起動時（またはレンダリング実行前）にPCのスペック（チップ種類・メモリ量）を自動検出し、利用可能なモデルの中から推奨モデルを提示する
- ユーザーは以下を個別に設定できる
  - プレビュー用モデル／本番書き出し用モデル（同一でも別でも可）
  - モデルごとの調整可能パラメータ（例: 品質・生成ステップ数・温度等、モデルの種類に応じて可変）
- レンダリングの進捗は、動画書き出しのようなプログレスバーで表示する
- **モーダル遮断型UIブロック**: 時間のかかる生成操作（Phase 1の作曲、モデルのダウンロード、Phase 2の本番書き出し）の実行時は、全画面をカバーするモーダルダイアログでプログレスバーを表示し、ユーザーによる他の操作を完全にブロックする。
- レンダリング中の**キャンセル**を可能にし、途中で中断してもアプリやOSがフリーズしないようにする（60分の本番レンダリングは数分〜十数分かかる可能性があるため、キャンセル導線は必須とする。ただし本番書き出し中はキャンセル不可とする）

### 6.7 長尺書き出し・ループ処理
- Phase 2の書き出しUI（書き出し形式、モデルティア、書き出し時間スライダー、無限ループオプション）は、Phase 1の作曲に成功するか、プリセットが正常にロードされた段階で有効化・表示される。
- 保存先は、macOS標準のファイル保存ダイアログ（`SAVE_FILE_DIALOG`）を呼び出してユーザーが自由に選択する。デフォルトの保存先ディレクトリとして、macOSの「ミュージック」フォルダ（`~/Music`）が自動で選択される。
- Phase 2でレンダリングした音声を、指定尺（1〜60分、6.2節のPCスペック上限に従う）までクロスフェード/ループでつなぎ合わせて書き出す。
- 出力形式: WAV（高音質）またはMP3（軽量）を選択可能。
- **無限ループ用に書き出し（オプション）**: チェックボックスをオンにすることで、ファイル両端のフェードイン・フェードアウトを無効化し、メディアプレイヤーやゲーム等で切れ目なくシームレスにループ再生できるBGMファイルを出力する。
- **音量正規化**: 書き出し時にラウドネス（例: LUFS基準）を一定範囲に揃え、モデル・パラメータの違いによる音量差を吸収する。
- **フェードイン/フェードアウト**: 通常書き出し時、曲の先頭・末尾に短いフェードを自動付与し、動画のBGMとして違和感なく開始・終了できるようにする。

### 6.8 プリセット保存
- 自然言語入力・スライダー値（曲の長さ含む）に加えて、**生成済みのMIDIデータ自体も含めて**プリセットとして保存できる
- 保存先はローカルファイル（`~/Library/Application Support/ABMM/presets/`等）で、MIDIファイル(.mid)とメタ情報(JSON)をセットで保存する
- プリセットを読み込むと、Phase 1をスキップしてそのMIDIデータから直接Phase 2（レンダリング）をやり直すことも可能

### 6.9 MIDIファイルのインポート/エクスポート
- Phase 1で生成されたMIDIデータを、アプリ専用のプリセット形式だけでなく、**標準MIDIファイル(.mid)としてエクスポート**できるようにする
  - GarageBand/Logic/Ableton等の外部DAWで開いて手を加えられるようにするための機能
- 逆に、**既存の.midファイルをインポート**して、Phase 2のレンダリング対象にしたり、Phase 1のたたき台として扱えるようにする
  - インポートしたMIDIは8章のJSONスキーマに変換し、`track_name`/`muted`等の項目にはデフォルト値を補完する
- これは14.1節で構想している「MIDIトラックエディタ／シーケンサーUI」の前提となる、ファイルの読み書き機能でもある

### 6.10 音楽知識のあるユーザー向け：DSL形式によるカスタムコード進行指定
- 自然言語入力欄において、特定の文脈（DSL記法）を用いてユーザーがコード進行や小節数、スタイルを直接指定できるようにする。
  - **DSL書式例**: `[Am7 -> D7 -> Gmaj7 -> Cmaj7 : 4 bars : style=jazz]`
- アプリケーションは自然言語入力からこのDSLパターン（例：`[...]`）を検出し、パースする。
- パースに成功した場合は、LLMによるコード進行決定フェーズをバイパス（または制約条件としてLLMに提供）し、指定された進行と小節数をPython側のMIDI音符展開エンジンにダイレクトに引き渡して、破綻のない正確なカスタム伴奏・フレーズを即座に組み立てる。

## 8. MIDIステップデータ スキーマ定義（Phase 1出力）

LLMが直接この形式のJSONを出力する。16分音符グリッドを基準とした「ステップ」で音符を表現し、
生MIDIバイナリよりもLLMが安定して生成しやすい形式にする。
将来のトラック編集機能（14.1節）を見据え、各トラックに`track_name`（表示名）と`muted`（オンオフ状態）を持たせておく。

```json
{
  "tempo_bpm": 82,
  "time_signature": "4/4",
  "key_mode": "major",
  "duration_minutes": 12,
  "sections": [
    {"name": "intro", "start_bar": 0, "bars": 4},
    {"name": "main", "start_bar": 4, "bars": 16},
    {"name": "outro", "start_bar": 20, "bars": 4}
  ],
  "tracks": [
    {
      "track_id": "track_1",
      "track_name": "Piano",
      "instrument": "acoustic_piano",
      "channel": 0,
      "muted": false,
      "notes": [
        {"step": 0, "pitch": 60, "velocity": 80, "duration_steps": 4},
        {"step": 8, "pitch": 64, "velocity": 70, "duration_steps": 4}
      ]
    },
    {
      "track_id": "track_2",
      "track_name": "Acoustic Guitar",
      "instrument": "acoustic_guitar",
      "channel": 1,
      "muted": false,
      "notes": [
        {"step": 64, "pitch": 55, "velocity": 60, "duration_steps": 2}
      ]
    },
    {
      "track_id": "track_3",
      "track_name": "Soft Drums",
      "instrument": "soft_drum_kit",
      "channel": 9,
      "muted": false,
      "notes": [
        {"step": 64, "pitch": 36, "velocity": 90, "duration_steps": 1}
      ]
    }
  ]
}
```

- `step`: 16分音符単位のグリッド位置（1小節=16ステップ、4/4拍子基準）
- `pitch`: MIDIノート番号
- `duration_steps`: ステップ単位の音の長さ
- `duration_minutes`: 曲全体の指定長さ（分）。生成時にセクション数・繰り返し回数を決める基準にする
- `track_id`/`track_name`/`muted`: 将来のトラック単位編集機能（オンオフ・並び替え・コピペ・改名）のための識別子と状態
- `sections`: 曲の構成（イントロ/展開/アウトロ等）を明示し、将来のシーケンサーUIでのセクション表示にも利用する
- このJSONは`pretty_midi`/`mido`で標準MIDIファイルに変換可能

## 9. LLMプロンプト設計（Phase 1: 作曲）

### 使用モデルとPCスペック推奨
- Ollamaで動作するオープンソースモデルを、PCの搭載メモリ（RAM）量に応じて動的に推奨・選択できるようにする。
  - **RAM 16GB - 24GB**: `llama3.2:3b`（軽量・最速）または `qwen2.5:7b` / `gemma2:9b`。低遅延でメモリを逼迫しないため、標準環境に最適。
  - **RAM 32GB - 48GB**: `qwen2.5:14b`、`gemma2:27b` または `mistral:7b`。指示追従性が高く、より複雑なアーティスト風アレンジや指示意図の理解に優れる。
  - **RAM 64GB以上**: `qwen2.5:32b` または `llama3:70b` (量子化版)。高度な作曲理論の応用、精緻な楽器バランスのプロンプト解釈が可能。
  - **RAM 128GB以上**: `llama3:70b` (FP16版) または `command-r-plus`。フルスケールのオーケストラ編成や複数セクションをまたぐ壮大なコード・構成アレンジ案の提示に最適。
- モデルはユーザーが設定画面から切り替え可能にする。

### システムプロンプト方針
- 「あなたは自然言語の曲の指示とパラメータから、指定JSONスキーマ（8章）に厳密に従ったMIDIステップデータのみを出力する作曲アシスタントです」と明示
- Few-shot例を2〜3個含め、出力フォーマットのブレを抑える
- 自然言語の自由記述（構成・雰囲気・楽器の指定）とスライダー数値（テンポ等＋**曲の長さ**）の両方を1つのプロンプトにまとめて渡す
- 曲が長い場合（`duration_minutes`が大きい場合）はセクション単位で分割生成し、後段で結合する方式を検討する（LLMの出力トークン制限対策）。もしくは短いセクションを生成し、ループ/バリエーション処理で尺を伸ばす方式との併用も検討する

### エラーハンドリング
- JSONスキーマ逸脱・パース失敗時は自動的にプロンプトを補正して再試行（最大2回）
- それでも失敗する場合はデフォルトの簡易テンプレートMIDI（4小節ループ）にフォールバックし、ユーザーに通知する

## 10. レンダリングモデルの選択・パラメータ設定（Phase 2）

3Dレンダリングや動画書き出しにおける「品質プリセット」と同様に、PCスペックや用途（プレビュー/本番）に応じてモデルとパラメータを選べるようにする。

### 10.1 モデルティア

| ティア | 想定モデル規模 | 推奨ハードウェア | 特徴 |
|---|---|---|---|
| 軽量（Lite） | SoundFontベースの合成（ニューラルモデル不使用） | 全Apple Silicon Mac | 最速。プレビュー用途に最適 |
| 標準（Standard） | 小型の音声生成モデル（数B規模、量子化済み） | M1〜、メモリ16GB程度〜 | 表現力とレンダリング時間のバランス型 |
| 高品質（High Quality） | 大型の音声生成モデル | M-Pro/Max/Ultra、メモリ32GB以上推奨 | 音質重視。レンダリング時間は長め |

### 10.2 プレビュー用モデルと本番用モデルの個別設定
- ユーザーは「プレビュー（数十秒試聴）で使うモデル」と「本番（全尺書き出し）で使うモデル」を個別に選択できる
- Phase 1（作曲）とPhase 2（レンダリング）も、同一モデルを共用する構成・別々の専用モデルを使う構成のどちらも取れるようにモデル管理を抽象化する
- モデルごとに調整可能なパラメータ（例: 品質レベル、生成ステップ数、温度、ランダムシード等）をUIから変更できるようにする。パラメータの種類はモデルの実装に応じて可変のため、モデルごとに設定項目を定義できる仕組みにする

### 10.3 ハードウェア検出と曲の長さ上限
- `platform`/`psutil`でOS・メモリ量を取得
- `sysctl -n machdep.cpu.brand_string`等でApple Siliconのチップ世代を判定
- 検出結果に基づき、
  - 動作可能なモデルティアの推奨を決定
  - **レンダリング可能な曲の長さ上限を自動算出**し、UIの長さスライダーの可動域に反映する
- 上限の算出ロジック（概要）: モデルティア・チップ世代・メモリ量から「1分あたりのレンダリング所要時間・メモリ使用量」を見積もり、実用的な待ち時間・メモリ制約に収まる上限を計算する（具体的な係数は実装フェーズで実機検証して決定する）

### 10.4 実装上の注意
- MIDI条件付きの音声生成を行うオープンソースモデルは現時点でエコシステムが発展途上のため、実装フェーズで具体的なモデルを比較検証しながら決定する
- レンダラーは共通インターフェース（`render(midi_data, model_tier, params) -> audio_waveform`）で抽象化し、将来的に新しいモデルが登場した際に差し替えやすい設計にする
- 軽量ティア（SoundFont合成）は常にフォールバックとして維持し、モデルのダウンロード前でも最低限の音声化・プレビューができるようにする

### 10.5 モデルのディスク容量管理
- Phase 1用モデル（Ollama）とPhase 2用モデル（複数ティア）を合わせると、ダウンロード済みモデルがディスク上で数十GB規模になり得るため、モデル管理画面を設ける
- 設定画面に、ダウンロード済みモデルの一覧・それぞれの使用容量・合計使用容量を表示する
- 使わなくなったモデルをアプリ内から削除できる導線を用意する（誤削除防止のため確認ダイアログを表示する）
- モデルダウンロード前に、空きディスク容量が不足していないかを確認し、不足している場合は警告を表示する

### 10.6 新モデルの動的追従（モデル定義ファイルのオンライン更新）
- 今後リリースされる新しいLLMモデルや音声合成モデル、高品位SoundFontに対し、アプリケーションの再コンパイル（バイナリ再ビルド）を必要とせずに追従できる動的構成管理システムを導入する。
- 起動時に、GitHub Releasesや特定のパブリックCDNに配置された `models_config.json`（最新モデル定義ファイル）を自動（または手動で）取得し、ローカルの定義リストをマージ・更新する。
- UI側のモデルセレクトボックスやモデル管理画面は、この動的定義ファイルを読み込んで描画するため、新しいモデル名、説明文、ファイルサイズ、ダウンロードURL、推奨スペック等の情報が即座にユーザーのインターフェースに反映される。
- ダウンロード済みの古いモデル構成も定義ファイル上で非推奨（Deprecated）のタグを付与することで、新規ダウンロードを無効化しつつ、ダウンロード済みのモデルは引き続きローカルで使用可能とするなど、後方互換性を保つ仕組みを提供する。


### 10.7 代表的オープンソース（OSS）モデル・ライブラリの適合性評価と採用方針

ABMMのコンセプト（Apple Silicon Mac完全ローカル動作、軽量・高速BGM生成、著作権フリー・インスト限定）に基づき、代表的なOSSモデル・ライブラリの評価結果および採用方針を以下のように定義・取り入れる。

| モデル・ライブラリ | カテゴリ | ABMMとの適合性 | 採用方針・適用内容 |
|---|---|---|---|
| **MidiTok** | シンボリックAI | ✅ **標準ライブラリとして採用** | 軽量なPython MIDIトークナイザーライブラリ。`requirements.txt` に導入し、Ollama等のローカルLLMとのMIDI表現相互変換・トークン効率化の基礎ツールとして組み込む。 |
| **FIGARO** | シンボリックAI | ⚠️ **構造設計の参考（非同梱）** | 構成・フレーズ制御に優れたMIDI生成モデルだが、大型PyTorch依存。ABMMはOllama標準LLMでの直接JSON/コードプラン生成を主軸とし、FIGAROの微細制御パラメータ（密度・展開など）のプロンプト設計のみ参考に採用。 |
| **Google Magenta (Music Transformer / MusicVAE)** | シンボリックAI | ⚠️ **非推奨** | 古いTensorFlow (1.x/2.x) 依存が多く、最新のPython 3.11+ / Apple Silicon macOS環境との互換性・保守性の観点から非推奨。 |
| **DDSP (Google Magenta)** | 音響合成AI | ⚠️ **将来拡張候補 (High Quality ティア)** | 物理モデル+ニューラルによる生楽器表現。Phase 2 レンダラー抽象化層（`BaseRenderer`）の将来的な追加プラグイン候補として位置づける。 |
| **Meta MusicGen (AudioCraft)** | 音響合成AI | ⚠️ **将来拡張候補 (ハイエンド)** | メロディ条件付けで高品質オーディオ生成が可能だが計算リソース/生成時間が非常に大。標準のFluidSynth（軽量・高速数秒再生）と切り離し、ハイエンド環境向け拡張機能として設計に盛り込む。 |
| **DiffSinger** | 音響合成AI | ❌ **不適合 (非採用)** | ボーカル/歌唱音声合成に特化したモデル。ABMMの「インストゥルメンタルBGM限定（ボーカルなし）」の基本仕様に反するため非採用。 |

## 13. 実装ロードマップ（vibe coding向けフェーズ分割）

### Phase A: MVP（Phase 1 作曲のみ）
1. 固定のシンプルなMIDI JSONスキーマ（トラック名・muted含む）を定義し、`pretty_midi`で.mid変換できることを確認
2. Ollama連携でLLMにMIDI JSONを生成させ、パース・バリデーションを実装
3. `pyfluidsynth`でPhase 1プレビュー再生ができることを確認

### Phase B: 自然言語＋スライダーの本実装
4. 自然言語入力＋スライダー値（曲の長さ含む）をプロンプトに統合するロジックを実装
5. スキーマ逸脱時のリトライ・フォールバック処理を実装

### Phase C: レンダリングフェーズ（軽量版）
6. `renderer_base`インターフェースを設計し、まずは`renderer_lite`（SoundFont合成）のみで一通り動作させる
7. ハードウェア検出ロジックを実装し、曲の長さ上限をUIスライダーに反映する

### Phase D: レンダリングフェーズ（ニューラルモデル・プレビュー機能）
8. ニューラル音声生成モデルの候補調査・比較検証
9. `renderer_neural`を実装し、軽量/標準/高品質のモデル切替を実際に動作させる
10. Phase 2の数十秒プレビュー機能（プレビュー用モデル/本番用モデルの個別選択含む）を実装

### Phase E: 仕上げ
11. 長尺書き出し（1〜60分）・クロスフェードループ処理
12. プリセット保存/読込（MIDI+メタ情報）
13. `.app`パッケージング（pyinstaller/py2app、Apple Silicon向けビルドのみ）

## 16. 非機能要件

- **Apple Siliconのみサポート**（Intel Macは非対応）
- 完全オフライン動作（ネットワーク接続不要。ただしモデル初回DL、GitHub連携の自動アップデート確認時を除く）
- Phase 1（作曲）は数秒〜十数秒程度で完了することを目標とする
- Phase 2プレビュー（数十秒レンダリング）は数秒〜十数秒程度で完了することを目標とする
- Phase 2本番レンダリングの所要時間はモデルティア・曲の長さに依存するが、軽量ティアは実用的な待ち時間で完了すること
- 曲の長さの上限（最大60分）は、PCスペックに応じて自動的に制限される
- アプリ単体で配布可能な`.app`バンドルとして提供（Python環境を別途要求しない）

### 16.1 プライバシー・通信ポリシー
- 自然言語入力・生成したMIDI/音声データ等のユーザーコンテンツは、外部サーバーに送信しない（テレメトリ・利用状況の収集は行わない）
- アプリが行う通信は、原則として以下に限定する
  - モデル（Ollama用LLM、Phase 2用レンダリングモデル）の初回ダウンロード
  - 14.3節の自動アップデート確認（GitHub Releases APIへの問い合わせ）
- 自動アップデート確認は設定画面からオプトアウト（無効化）できるようにする
- どのような通信が発生するかをREADME等のドキュメントに明記し、ユーザーが把握できるようにする

## 17. ビルド・配布パイプライン

アプリの配布形態は、**PyInstaller（または同等のバンドルツール）でmacOSの`.app`（AppBundle）を作成し、
GitHub Actionsでビルド・ZIP化・配布まで自動化する**方式とする。

### 17.1 ビルドツール
- **PyInstaller**を第一候補とする（同等の代替として`py2app`も選択肢だが、依存パッケージの解決しやすさから
  PyInstallerを推奨）
- ビルド定義はリポジトリ内に`.spec`ファイルとして保持し、以下を設定する
  - エントリポイント: `app/main.py`
  - 同梱リソース: `frontend/`以下のHTML/CSS/JS一式、将来的にはSoundFont等のアセットも同梱
  - ターゲットアーキテクチャ: `arm64`固定（3章の通りApple Siliconのみサポートのため）
  - `BUNDLE()`定義で`.app`化し、`Info.plist`にアプリ名・バージョン・最小対応OSバージョン等を設定する

### 17.2 GitHub Actionsによるビルド自動化
GitHub Actions上のワークフローで、以下の流れを自動実行する。

1. **トリガー**: `main`ブランチへのpush、リリース用タグ（例: `v*`）のpush、Pull Request、および手動実行（`workflow_dispatch`）
2. **実行環境**: Apple Silicon（arm64）ネイティブのGitHub-hostedランナー（`macos-14`以降のイメージ）を使用する。
   Intel(x86_64)ランナーは本アプリの対応OS（3章参照）に合わないため使用しない
3. **セットアップ**: リポジトリのチェックアウト、Python（arm64版）のセットアップ、`requirements.txt`に基づく依存パッケージのインストール
4. **ビルド**: `pyinstaller <spec file> --noconfirm --clean`を実行し、`dist/`配下に`.app`を生成する
5. **コードサイニング**: Apple Developer ID証明書を使わない場合は、少なくとも**ad-hoc（自己署名）**でコードサイニングを行う
   （`codesign --force --deep --sign - <App.app>`）。これにより一部の実行時警告を軽減できるが、
   Gatekeeperの完全な承認（配布先での警告なし起動）にはApple Developer IDによる正式な署名と
   notarizationが別途必要になる（17.4節参照）
6. **ZIP化**: `.app`をそのまま配布するとmacOSのGatekeeper属性やリソースフォークが壊れやすいため、
   `ditto`コマンド（`ditto -c -k --sequesterRsrc --keepParent`）でZIP化してから配布物とする
7. **配布**:
   - 通常のpush/PRビルドでは、GitHub Actionsの**Artifacts**としてZIP化された`.app`をアップロードし、
     開発中の動作確認に使えるようにする
   - リリース用タグ（`v*`）のpush時は、**GitHub Releases**を自動作成し、ZIP化された`.app`を
     リリースアセットとして添付する

### 17.3 配布物の利用方法（想定）
- ユーザーはGitHub ReleasesページからZIPファイルをダウンロードし、解凍して`.app`をApplicationsフォルダ等に配置して利用する
- ad-hoc署名のみの場合、初回起動時にGatekeeperの警告が表示されるため、README等で
  「右クリック→『開く』」による回避手順を案内する

### 17.4 将来の課題（正式配布に向けて）
- Apple Developer IDアカウントを取得し、Developer ID Application証明書によるコードサイニングを導入する
- `notarytool`等を用いたApple公証（notarization）をGitHub Actionsワークフローに組み込み、
  配布先でGatekeeper警告が出ないようにする
- 証明書・Apple IDパスワード（App用パスワード）等の機密情報はGitHub Actionsの`secrets`で管理する
- 自動アップデート機能（14.3節）とあわせて、署名済み・公証済みのビルドを配布できる状態を目指す
